# Today I Learned (2026-05-04)

## 테스트 (E2E·TDD·테스트하기 쉬운 코드)

- E2E 테스트는 버튼 클릭·페이지 이동·입력 같은 실제 사용자 동작을 브라우저에서 흉내 내 앱을 처음부터 끝까지 검증하는 방식이다. 유닛 테스트가 개별 함수만 본다면, E2E는 화면 전환·세션 유지처럼 여러 요소가 함께 동작하는 흐름의 문제를 잡아낸다.
- TDD는 실패하는 테스트를 먼저 쓰고(Red) → 통과할 최소 코드를 짜고(Green) → 리팩토링하는(Refactor) 사이클로 개발하는 방법론이다. 테스트가 안전망이 되어 디버깅 시간이 줄고 리팩토링에 자신감이 생긴다.
- 순수 함수·단일 책임·예측 가능한 코드일수록 테스트하기 쉽다. 반대로 테스트가 어렵다면 보통 함수가 너무 많은 일을 하거나 의존성이 강하게 얽혀 있다는 설계 신호다.

## 결합도와 응집도

- 결합도는 모듈끼리 서로 얼마나 의존하는지, 응집도는 한 모듈 안의 요소들이 한 목적으로 얼마나 모여 있는지를 뜻한다. 좋은 설계는 결합도를 낮추고 응집도를 높이는 방향이며 둘은 보통 반비례한다.
- 인터페이스를 통해 의존하면 결합도를 낮출 수 있고, 한 파일이 인증과 DB 처리를 같이 떠안으면 응집도가 낮아진다.

## 느린 API·외부 라이브러리 버그 대응

- API가 느릴 때는 먼저 응답 시간 수치를 확보해 "특정 API가 평균 3초"처럼 구체적으로 소통하는 게 중요하다. 단기간에 못 고치면 프론트에서 로딩 UI·prefetch·캐싱(React Query 등)으로 체감 지연을 줄인다.
- 외부 라이브러리 버그는 버전 변경·특정 컴포넌트 여부로 원인을 좁히고, 임시로 스타일을 덮거나 대체 구현으로 우회한다. 길어지면 이해관계자에게 공유하고, 장기적으로는 리포트·PR·포크로 대응한다.

## 쿠키와 세션

- 쿠키는 클라이언트에 저장돼 요청마다 자동으로 함께 전송되는 데이터이고, 세션은 서버에 상태를 저장한 뒤 세션 ID만 주고받는 방식이다.
- 인증에는 자동 전송 + `HttpOnly`로 XSS를 막을 수 있는 쿠키를 주로 쓴다. 강제 로그아웃이나 동시 접속 제한처럼 서버가 로그인 상태를 직접 쥐어야 할 때는 세션이 유리하다.
